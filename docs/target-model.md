# embedded-text target model

## Goal

Move the library toward a simple generated-data + drawable-runtime split.

## Runtime data model

`FontData` is the only object generated by macros/codegen. It represents one pixel size.

`FontData` contains:

- string index data stored directly as `&'static str`.
  - The string is the ordered character index for glyph lookup.
  - Each character in the string maps to the glyph at the same character position.
- `char_size: usize`.
  - One `FontData` is one size only.
  - Different sizes must be generated as different `FontData` values / macro invocations.
- bitmap byte array reference.
- glyph array reference.

`Glyph` contains per-character bitmap metrics:

- `bitmap_offset`
- `width`
- `height`
- `x_offset`
- `y_offset`
- `x_advance`

`y_offset` stays in `Glyph` because different fonts inside one size can need different vertical offsets.

## Drawable runtime object

Create `DrawableText` on top of `FontData`.

`DrawableText` contains:

- `font_data: &FontData`
- text to render: `&str`
- ASCII cell size: `embedded_graphics::geometry::Size`
- non-ASCII / Chinese cell size: `embedded_graphics::geometry::Size`
- render start point: `embedded_graphics::geometry::Point`
- foreground color

The runtime uses ASCII/non-ASCII cell sizes to advance the pen. It should not encode layout policy inside generated font data.

## Macro/codegen responsibility

Macros only create `FontData`.

A macro invocation creates one size only, but may include multiple font files for the same size. Each font can provide its own y offset parameter; that offset is baked into the generated glyph data.

Expected direction:

```rust
font_data! {
    pub static FONT_16: FontData<'static> = {
        size: 16,
        fonts: [
            { path: "assets/ascii.ttf", glyphs: "ASCII...", y_offset: 0 },
            { path: "assets/cjk.otf", glyphs: "你好世界", y_offset: -2 },
        ]
    };
}
```

No macro should create `DrawableText`; text rendering state belongs to runtime user code.

## Compatibility note

The previous `BitmapFont` / `BitmapText` names are being corrected toward `FontData` / `DrawableText`. Temporary aliases may exist during migration, but new code should use the new names.
